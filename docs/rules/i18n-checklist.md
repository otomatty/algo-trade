# 国際化対応（i18n）チェックリスト

## 概要

新規コンポーネント作成時に、国際化対応が正しく実装されているかを確認するためのチェックリストです。

## 新規コンポーネント作成時のチェックリスト

### 1. 基本実装

- [ ] `useTranslation()` フックをインポートしている
- [ ] 適切な名前空間を指定して `useTranslation()` を呼び出している
- [ ] ハードコードされたテキストが存在しない
- [ ] 全てのユーザー向けテキストが翻訳キーを使用している

### 2. 翻訳ファイル

- [ ] 日本語翻訳ファイル（`src/i18n/locales/ja/{namespace}.json`）を作成・更新している
- [ ] 英語翻訳ファイル（`src/i18n/locales/en/{namespace}.json`）を作成・更新している
- [ ] 翻訳キーが両方の言語ファイルに存在する
- [ ] 翻訳キーの構造が一致している（JSON構造が同じ）

### 3. i18n設定

- [ ] `src/i18n/index.ts` の `resources` に新しい名前空間を追加している
- [ ] `src/i18n/index.ts` の `ns` 配列に新しい名前空間を追加している
- [ ] 型定義ファイル（`src/i18n/resources.ts`）を更新している（必要に応じて）

### 4. 翻訳キーの品質

- [ ] 翻訳キーが camelCase で命名されている
- [ ] 翻訳キーが意味的に明確である
- [ ] 階層構造を適切に使用している（例: `buttons.save`, `labels.loading`）
- [ ] 既存の `common` 名前空間の翻訳キーを再利用できる場合は使用している

### 5. 名前空間の選択

- [ ] 適切な名前空間を選択している
  - `common`: 共通UI要素（ボタン、ラベル、メッセージなど）
  - `navigation`: ナビゲーション関連
  - 機能別名前空間: 機能固有の翻訳（例: `dashboard`, `data-management`）
- [ ] 複数の名前空間を使用する場合は、適切に使い分けている

### 6. テスト

- [ ] テストファイルで `I18nextProvider` を使用している
- [ ] テストで翻訳キーの存在を確認している
- [ ] テストで言語切り替えを確認している（必要に応じて）
- [ ] テストが全て通過している

### 7. コードレビュー観点

- [ ] ハードコードされたテキストがないか確認
- [ ] 翻訳キーの命名規則に従っているか確認
- [ ] 名前空間の使用が適切か確認
- [ ] 翻訳ファイルの構造が適切か確認
- [ ] 既存の翻訳キーを再利用できる場合は使用しているか確認

## よくあるミス

### ❌ 悪い例

```typescript
// ハードコードされたテキスト
<Button>保存</Button>

// 名前空間なし
const { t } = useTranslation();

// 不適切な翻訳キー名
{t('text1')}

// 翻訳ファイルの不一致
// ja/common.json: { "save": "保存" }
// en/common.json: { "saveButton": "Save" } // キー名が違う
```

### ✅ 良い例

```typescript
// 翻訳キーを使用
const { t } = useTranslation('common');
<Button>{t('buttons.save')}</Button>

// 適切な名前空間の選択
const { t } = useTranslation('dashboard');

// 意味的な翻訳キー名
{t('dashboard.title')}

// 翻訳ファイルの一致
// ja/common.json: { "buttons": { "save": "保存" } }
// en/common.json: { "buttons": { "save": "Save" } }
```

## チェックリストの使い方

1. 新規コンポーネント作成時、このチェックリストを確認
2. 各項目をチェックしながら実装
3. コードレビュー時に、このチェックリストを使用して確認
4. 全ての項目にチェックが入るまで実装を完了しない
